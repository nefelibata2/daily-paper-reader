# SPARK: Security Knowledge Priming and Representation-guided Knowledge Activation for LLM-based Secure Code Generation 

Xiaoyun Xu 1, Lichao Wu 2, Jona te Lintelo 1, Siyu Zhang 2, Stjepan Picek 3,11Radboud University, 2University of Bristol, 3University of Zagreb 

{xiaoyun.xu,jona.telintelo,stjepan.picek }@ru.nl, {lichao.wu,s.zhang }@bristol.ac.uk 

Abstract —Large language models routinely generate code with exploitable security flaws. Prior literature attributes this limi-tation to a lack of security expertise, steering current defense mechanisms toward heavy fine-tuning or external knowledge retrieval, which introduces significant computational overhead and data bias through redundant code examples. Contrary to this view, we argue that pretraining corpora are already rich in security material. The bottleneck is activation: without an explicit and brief cue, statistical pressure toward common training-distribution patterns suppresses the model’s safety-relevant representations. We present SPARK, an inference-time security harness that activates this latent knowledge without any retraining. The harness has two parts. Component Iretrieves a few of the relevant Common Weakness Enumeration (CWE) entries for each coding task and appends a short structured cue to the prompt; this alone is enough to surface the model’s existing security representations. Component II adds a precomputed token bias to the logits at every decoding step. We obtain the bias by projecting a safe-direction vector, the unit difference between the mean safe and mean unsafe last-layer hidden states, through the language model head. The bias is computed once offline; applying it costs a single vector addition per generated token. We evaluate SPARK on 9 open-source models across C++, Java, and Python, and compare with 7 baselines spanning fine-tuning and retrieval-augmented methods. SPARK matches or improves on the best baseline in every setting while preserving HumanEval utility. We further test Component I in a black-box setting on 7 of today’s strongest models, including Claude, DeepSeek, and GPT, demonstrating the bottleneck of insecure code generation and the improvements enabled by our method.

Limitations of existing approaches. Prior work has largely overlooked this insight, instead treating the problem as a knowledge deficit to be remedied through expensive inter-ventions. Fine-tuning-based methods , such as SVEN [11], SafeCoder [12], and GoodVibe [32], retrain or modify model parameters, overwriting security knowledge the model al-ready possesses and requiring a model-specific training pipeline for every new backbone. Retrieval-augmented methods , such as RESCUE [30] and CodeGuarder [18], inject large quantities of external code examples into the prompt, filling up the context with information that the model could derive from its own representations. Both strategies are effective but introduce additional complexity through model retraining or large prompt augmentation. If the bottleneck lies in how security knowledge is incorpo-rated at inference time rather than in its absence, what is needed is a targeted trigger, not additional training or redundant context. 

Our goal and contributions. We present SPARK (Security knowledge Priming And Representation-guided Knowledge activation), an inference-time security harness for LLM-based code generation: a training-free, model-agnostic method that activates the model’s latent security knowl-edge without modifying its parameters or injecting external code examples. SPARK builds on 2 components, Com-ponent I (Figure 2) : at inference time, it retrieves the CWE entries [21] most relevant to the coding task and appends a concise structured summary to the prompt. Unlike RESCUE [30] and CodeGuarder [18], it supplies no new code examples; it explicitly names the relevant vulnerability class, cueing the model to draw on its existing security representations. To check whether this cue activates the model’s latent security knowledge, we introduce a safe direction δ in the model’s representation space, the direction from the unsafe code to the safe code representation, and use it as a proxy for where security knowledge lies. We find that Component I alone, with no change to decoding, shifts the model’s pre-generation representation toward δ (see Figure 4 in Section 4.5), confirming that the cue activates security knowledge rather than supplying it. Building on this verifica-tion, we repurpose the same safe direction δ as an additional module, Component II (Safety-constrained decoding): at every autoregressive step, we add a precomputed token bias obtained by projecting δ through the language model head, steering generation along the safe direction. Reusing the verification signal directly during decoding gives a further improvement in secure code generation, at negligible per-token cost and without any parameter update. Existing studies have focused mainly on open-source models of modest scale, leaving the security be-havior of flagship models comparatively underex-amined, while they also produce a large fraction of insecure code.  

> •

We propose a new perspective on secure code gen-eration: rather than injecting external knowledge through large security knowledge bases or altering model weights through fine-tuning, we activate the security knowledge already encoded in pre-trained models with a lightweight inference-time cue. We further verify that this activation genuinely occurs. Using a safe direction in the model’s representation space as a proxy for its internal security knowledge, we show that the cue alone, with no change to de-coding, shifts the model’s representation toward this direction, and we then repurpose the same direction as a decoding-time module that further improves secure generation.

## 2. Background & Related Work

M generates a response y = ( y1, . . . , y T ) by autoregres-sively sampling each token yt ∼ pθ (· | qi, y <t ). At each decoding step t, the model computes a last-layer hidden state ht ∈ Rd and projects it through the language model head Wlm ∈ R|V |× d to obtain raw logits:

2.2. Coding LLM and Vulnerabilities 

AI code generation has achieved great success in the past few years. Flagship proprietary models, led by Ope-nAI’s GPT-5.4/5.5 [24, 26] and Anthropic’s Claude Son-net [1], represent the frontier of code intelligence. Lever-aging massive compute and advanced inference-time rea-soning, these models excel at complex algorithmic synthe-sis and repository-scale software engineering benchmarks. In the meantime, the open-source community has fostered highly competitive coding models. Early foundations like CodeLlama [29] established localized code synthesis frame-works by extending general-purpose architectures. Build-ing on this paradigm, recent architectures have achieved substantial performance breakthroughs. For instance, the Gemma 3 [7] and Gemma 4 [9] series from Google provide lightweight yet highly performant multi-modal reasoning baselines optimized for constrained computing resources. Concurrently, specialized series such as Alibaba’s Qwen 3 [36, 28] leverage expansive pre-training corpora (e.g., up to 36 trillion tokens) and hybrid-thinking modes to achieve state-of-the-art agentic coding capabilities. However, LLM-generated code does not confirm its security. Pearce et al. [27] conducted a systematic study of security weaknesses in GitHub Copilot outputs, finding that 40% of generated programs contained CWE-classified vulnerabilities across a diverse set of coding scenarios. Subsequent work has replicated and extended this finding to ChatGPT and other instruction-tuned models [16, 33], confirming that the problem is not an artifact of any single system. Moving beyond the synthetic queries utilized in prior evaluations, Khanmohammadi et al. [15] introduce WildCode, a large-scale empirical analysis of real-world, ChatGPT-generated code. Their findings further confirm existing concerns regarding the security vulnerabilities in-herent in LLM outputs. In addition, it is also difficult for LLM to conduct real-time vulnerability fixing. Due to the autoregressive architecture, the early defect could introduce complex downstream failures [23, 37]. 

2.3. Fine-tuning-based Defences 

Fine-tuning approaches improve the security of code-generating models by updating their parameters on a curated dataset of secure and vulnerable code pairs. The underlying assumption is that exposing a model to enough correc-tive examples causes its internal representations to shift toward safer generation patterns, making the improvement persistent across all subsequent queries without any runtime overhead. This paradigm has attracted considerable research interest, and its methods divide broadly into two categories: parameter-efficient adaptation and full network updates. Full-parameter fine-tuning modifies every weight in the model using a security-oriented objective and can yield strong gains, but is more likely to risk catastrophic forgetting when the training distribution is narrow [31]. Parameter-efficient alternatives such as LoRA [13] mitigate this by restricting updates to low-rank adapter matrices, yet they still require a representative training corpus, an optimization run, and full access to the model’s weights. Within this general framework, several methods have been proposed with more targeted objectives. SVEN [11] trains security-prefix vectors under a contrastive objec-tive, prepending a learned prefix at inference time to bias generation toward safe completions without modifying the backbone weights. SafeCoder [12] constructs a fine-tuning corpus from vulnerability-fixing commits mined from open-source repositories, training the model to prefer the cor-rected version of each change. GoodVibe [32] takes a more targeted approach by identifying security-relevant neurons through gradient attribution and restricting parameter up-dates to those neurons, reducing interference with unre-lated capabilities while focusing the adaptation budget on security-critical pathways. However, fine-tuning style meth-ods share a fundamental constraint: each new model family requires a separate fine-tuning run. 

2.4. Prompt-based and Retrieval-based Defenses 

Prompt-based and retrieval-augmented generation (RAG) methods improve code security without modifying model parameters, instead shaping generation by changing what the model receives as input. Since they require no training and impose no constraint on the model itself, these approaches can in principle be applied to any AI coding system at inference time and updated independently of the underlying weights. The simplest instantiation is a security-oriented system prompt that instructs the model to produce vulnerability-free code. PromSec [22] pursues greater specificity through iter-ative prompt optimization: it combines a graph generative adversarial network (gGAN) with static security analysis to iteratively refine the prompt until the generated code falls below a target CWE count, achieving notable reductions in vulnerability rates while preserving functional correct-ness. Retrieval-augmented approaches address the speci-ficity problem differently, by dynamically fetching relevant content from a curated knowledge source and prepending it to the query. RESCUE [30] retrieves semantically sim-ilar secure code snippets from a curated repository and prepends them to the prompt as in-context examples. Code-Guarder [18] extends this idea by constructing a large-scale vulnerability knowledge base that broadens the coverage of retrievable security patterns. CoSec [17] takes a different route to retrieve knowledge from a separate LLM. It employs a dedicated, smaller security model to assess the safety of generated code at runtime and flag unsafe outputs. 

## 3. The SPARK Security Harness 

SPARK is an inference-time security harness composed of two orthogonal components that collectively activate the LLM’s latent security competence during code generation. Component I (Section 3.2) operates at the prompt level, injecting a CWE-derived activation signal before generation begins. Component II (Section 3.3) operates at the decoding level, enforcing a continuous bias toward safe token choices at every autoregressive step. Neither component modifies model parameters; together they form a transparent layer that can be placed around any instruction-following LLM. 

3.1. Threat Model 

Goal. The defender aims to reduce the rate at which a de-ployed LLM produces code containing exploitable security weaknesses without modifying model parameters. Retrain-ing is excluded due to the cost and the risk of degrading general coding ability. 

Knowledge. We consider two access levels that reflect com-mon deployment conditions. In the black-box setting, the defender observes only the model’s text outputs and has no visibility into weights, hidden states, or logits; this is the standard condition when using a commercial model through a public API. In the white-box setting, the defender has full access to model parameters and can inspect intermediate computations at every layer and decoding step; this applies when an organization runs an open-source model locally. 

Capabilities. The two access levels translate directly into different intervention points. A black-box defender can only control what the model receives as input. Component I is designed for this setting: it retrieves relevant CWE entries and appends a structured activation cue to the prompt before the model call, requiring no access beyond standard text input and output. Our evaluation of Claude, DeepSeek, and GPT via their respective APIs operates under this constraint. A white-box defender can additionally intercept and modify the raw logits at each autoregressive step. Component II exploits this capability: it injects a precomputed token safety bias into the logit distribution at every decoding step, con-tinuously steering generation toward safer token choices. When white-box access is available, the two components are applied together. Component I establishes security-relevant context at the prompt level before generation begins; Com-ponent II reinforces the constraint at every subsequent de-coding step. 3.2. Harness Component I: Security Knowledge Activation 

Component I activates the model’s existing security knowledge by providing an explicit, task-specific cue about which vulnerability class is relevant. Otherwise, the model’s latent security representations are suppressed by the statis-tical prior toward common coding patterns. Unlike conven-tional RAG frameworks that require building extensive ex-ternal knowledge bases [18] and designing intricate chunk-ing or retrieval strategies [30], our approach bypasses such architectural overhead. We leverage concise, static CWE descriptions to awaken the model’s internal expertise. We address this by appending a structured CWE-derived context block to the generation prompt as a suffix, ensuring it is the last information the model processes before producing the first output token and thereby maximizing its causal influence on generation [20]. 

Knowledge Base Construction. We maintain a knowledge base built on the NIST Common Weakness Enumeration (CWE) catalog. For each CWE entry, we store the iden-tifier, a short name, a natural-language description, a list of concrete mitigations, and language-tagged code exam-ples illustrating vulnerable ( Bad ) and corrected ( Good ) pat-terns. All entries are pre-encoded into dense vectors using a sentence-transformer encoder ϕ : T → RD (specifi-cally all-MiniLM-L6-v2 ) and stored in an offline index

## 4. Experimental Analysis 

4.1. Experimental Setup 

Models. As listed in Table 2, we evaluate on nine open-source instruction-following code LLMs, including six mod-els, CodeLlama-7B-Instruct, Llama-3-8B-Instruct, Qwen3-[8B, 14B], and Gemma-3-[4B, 12B] for the main evalua-tion and three larger models, Qwen3-32B, Qwen3.6-27B, and Gemma-4-31B, for the model-scaling analysis in Sec-tion 4.8. All models are loaded in bfloat16 precision. In ad-dition, we also evaluate SPARK Component I in black-box on seven of the latest commercial models from Anthropic, DeepSeek, and OpenAI via API access. 

Dataset. We use CyberNative Code Vulnerability and Secu-rity DPO Dataset [3], a paired dataset of (question, safe im-plementation, unsafe implementation) triples spanning C++, Java, and Python. Entries are filtered by target language to form per-language evaluation pools. The same 80/20 train/eval split with a fixed random seed is used across all experiments: the training split provides the labeled pairs for computing the safe direction vector in Component II, and the evaluation split is held out for all security metrics.

4.4. Utility Preservation 

HumanEval [2] is a hand-written benchmark of 164 programming problems, each consisting of a function sig-nature, a docstring, and a set of unit tests. A model is given the function header and docstring and asked to complete the implementation; correctness is measured by pass@ k,the probability that at least one of k sampled completions passes all unit tests. We use pass@1 under greedy decoding throughout. HumanEval+ [19] extends the original suite with a substantially larger set of automatically generated test cases per problem, exposing edge cases that the original tests overlook and providing a stricter estimate of functional correctness. Figure 5 shows that SPARK preserves functional correct-ness on all but the smallest model. On HumanEval, SPARK changes pass@1 by only +0 .1% on average across 5 mod-els. We also notice a single outlier: there is a clear drop on CodeLlama-7B. We attribute this drop to CodeLlama-7B’s limited baseline capability: it has the lowest HumanEval pass@1 among the evaluated models, and as a smaller model used zero-shot without any task-specific fine-tuning, its instruction following is less robust. The CWE context that SPARK prepends acts as an additional instruction layer, which such a model struggles to satisfy without sacrificing the primary coding task. Stronger instruction-tuned models receive the same context with no measurable utility loss. Relative to the baselines, SPARK’s utility cost is small and is achieved without any gradient update or stored check-point. Fine-tuning methods incur larger and less predictable drops: SVEN loses the most ( −7.5% on HumanEval), LoRA averages −4.0% with high variance across models, and only GoodVibe is roughly utility-neutral. Retrieval-augmented baselines unexpectedly impair coding ability as well, with RESCUE and CodeGuarder each losing 6 to 7% on average and far more on the smaller models, because they append unfiltered vulnerability examples and mitigation text that crowd out the task. 4.6. Ablation Study 

Component Contribution. To measure the additive value of each harness component, we evaluate 4 conditions across all 3 languages: (i) baseline with no harness, (ii) Component I only (CWE prompt augmentation), (iii) Component II only (logit bias), and (iv) the full SPARK harness. As shown in Table 6, component I, which prepends concise CWE descriptions and mitigation cues to the gen-eration prompt, is the primary mechanism through which SPARK achieves security improvement. Rather than intro-ducing information the model has never seen, the structured CWE prompt elicits security knowledge already encoded in the model weights during pretraining. This interpretation is consistent with the magnitude of the gains: CodeLlama-7B improves from a near-zero baseline of 2.1% to 71.1% on C ++ and from 40.8% to 97.2% on Java solely through prompt augmentation, without any parameter update. That these gains come from a lightweight, inference-time inter-vention (rather than fine-tuning) suggests that capable code models already possess substantial latent security knowledge that standard prompts fail to bring out. In addition, to test whether Component I’s gains come from task-relevant re-trieval rather than from any security cue, we keep the gener-ation pipeline, test split, K, and decoding settings fixed and replace the retrieved CWEs with K entries drawn at random from the knowledge base (restricted to CWEs that have a target-language example, so that only relevance differs). The retrieved cue consistently outperforms the random cue by 20.7% on average (see Table 9 in Appendix F), confirming that matching CWEs to the coding task, rather than merely prepending security text, drives the improvement. Component II uses the local paired dataset to compute a safe-direction vector in the model’s representation space, producing a data-driven logit bias that complements the prompt-level signal of Component I. Stacked on top of Component I, it adds consistent if modest gains in 5 of the 6 conditions, with the largest synergy on CodeLlama-7B C ++ , a further 9.9% over Component I alone. Applied in isolation, however, without the prompt foundation of Component I, Component II yields results indistinguishable from the unmodified baseline, confirming that it refines an already security-oriented generation distribution rather than creating one. This asymmetry follows from the two com-ponents acting at different levels of the generation process: Component I conditions the model on vulnerability-specific knowledge before any token is produced, shaping the overall intent of the output, whereas Component II pushes the per-token logit distribution toward the safe subspace identified from the paired examples. The combination is therefore most effective when Component I has already established a safe-leaning context for Component II to reinforce.

To examine how SPARK interacts with model capac-ity, we evaluate the Qwen3 and Gemma families across 3 languages as the parameter count increases. Figure 9 plots Safe Code Rate as a function of model size for each family and language, with the baseline and SPARK curves shown separately. The shaded gap between them quantifies the benefit conferred by the harness at each scale point. Across both families, SPARK consistently exceeds the baseline at every size point tested. For the Qwen3 family, the absolute gain is largest at the 2 smallest models: Qwen3-8B and Qwen3-14B, each improves by approximately 41% on C ++ and by roughly 31% on Java and Python. As model capacity grows, the baseline rises substantially. Qwen3-32B reaches 66.9% on C ++ , and Qwen3.6-27B reaches 80.3%. SPARK continues to add a meaningful margin of 14.8% and 14.8% respectively on C ++ , and reaches above 95% on all 3 languages for Qwen3.6-27B. The pattern suggests that the CWE-guided prompt activates a growing reservoir of latent security knowledge as model capacity increases, rather than providing a fixed additive offset. The Gemma family exhibits a different scaling profile. Gemma-3-12B has a strong C ++ baseline of 81.0% but collapses to 40.1% on Java and 17.6% on Python, a spread of 63.4% within a single model. SPARK substantially reduces this cross-language variance: it raises the Java rate to 66.9% and the Python rate to 77.5%, narrowing the gap to 23.9%. Gemma-4-31B largely resolves the language imbalance at the baseline level, and SPARK further improves it. Across both families, the diminishing absolute gain at large scales does not indicate reduced effectiveness; rather, it reflects a ceiling effect as baselines approach the upper range already secured by SPARK at smaller sizes. 

## 5. Conclusion 

This paper argues that insecure code from modern LLMs is primarily an activation problem rather than a knowledge deficit. SPARK acts on this view as a training-free inference-time harness: Component I appends a structured CWE sum-mary to surface the model’s existing security representa-tions, and Component II adds a precomputed token safety bias derived from a small set of paired examples. Both com-ponents add negligible overhead per query. Across 6 open-source models and three popular coding languages, SPARK matches or exceeds the strongest fine-tuning, prompt, and retrieval baselines on every model-language pair while preserving HumanEval utility. Component I alone trans-fers to closed-source APIs from Anthropic, OpenAI, and DeepSeek, raising safe code rates by tens of percentage points in a fully black-box setting. Methods that retrieve external corpora or retrain the backbone accept a coverage ceiling bounded by their auxiliary resource. SPARK replaces it with the model’s own pretraining knowledge, the largest security resource available at inference time.

## Appendix A. Ethics Considerations 

This work develops a defensive technique that reduces the rate at which language models generate insecure code. Its primary effect is to mitigate, rather than create, security risk. All experiments use publicly released datasets and models. The CyberNative Code Vulnerability Security DPO and HumanEval datasets and the open-source models are used under their respective licenses, and the commercial models are accessed through their official APIs in accor-dance with the providers’ terms of service. No private, proprietary, or personally identifiable data is used at any stage. Our evaluation involves generating and analyzing code that contains known classes of vulnerabilities (e.g., CWE-classified weaknesses). All such code is produced and ex-amined solely within an isolated research environment; it is never executed against, or deployed to, any real system, and we release no new exploit or previously unknown vulnera-bility. The illustrative examples in the paper correspond to well-documented weakness classes. 

## Appendix B. Discussion: Limitations and Future Work 

Limitations. This work has two primary limitations. First, our evaluation assumes benign, cooperative users. We have not tested SPARK against adversarial prompts designed to bypass the security harness. An adversary could ex-plicitly instruct the model to ignore the CWE activation cue (e.g., “disregard all security advice”) or to produce vulnerable code in disguised forms. Whether Component I and Component II remain effective under such adversarial inputs is unknown. Second, our experiments are confined to single-file, self-contained code generation tasks. Real-world deployments often involve larger, multi-file repositories or 

agentic workflows where an LLM iteratively edits existing code, navigates across files, and maintains persistent state. In such settings, security vulnerabilities may span multiple functions or files, and SPARK’s per-query activation cue and per-step logit bias may not sufficiently propagate safety constraints across long, interleaved interactions. 

Future Work. To address these gaps, we plan two main extensions. First, we will evaluate SPARK under adversar-ial prompts, including instruction overriding, role-playing, and context poisoning attacks, and develop countermeasures such as prompt sanitization or adversarial training of the bias vector. Second, we will adapt SPARK to repository-level and agentic scenarios by designing a persistent security context (e.g., cross-file CWE tracking) into agent loops with memory of prior unsafe patterns.

## Appendix E. Time Consumption 

Table 8 reports the total generation wall-clock time over the 142-sample evaluation under each condition. Compo-nent II adds no measurable decoding overhead: because it applies only a fixed per-token bias to the logits, its runtime tracks the Baseline that uses the same prompt, and the full SPARK (Component I + II) tracks Component I almost ex-actly. The added cost of SPARK over the Baseline therefore comes entirely from Component I, whose CWE-augmented prompt lengthens the input and thus the prefill and decoding time. The safe direction used by Component II is computed once offline in a few seconds per model and does not affect inference. Overall, SPARK’s security gains are obtained at the cost of a longer prompt, with the decoding-level intervention essentially free. Pre-computing the direction bias costs only seconds for each case, and it is computed once, so we do not include it.
