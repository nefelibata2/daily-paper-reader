## 摘要
大型语言模型（LLMs）正逐渐从独立的文本生成系统发展为调用外部工具、访问环境并执行多步任务的智能体。然而，传统的函数调用基准主要评估任务完成度和API正确性，而隐私评估基准则通常关注最终响应或隐私判断。这两种视角都无法捕捉执行多工具轨迹过程中基于目的的信息流动。受当前智能体评估的局限所启发，ToolPrivacyBench审计任务私有原子是否仅被路由至授权工具和下游接收方，从而同时评估工具使用过程中的任务完成情况和隐私过度披露。该基准包含2150个案例，其中1150个为完全合成的隐私敏感业务工作流，1000个改编自现有的多工具和函数调用基准。每个案例均由一个策略知识库表示。在智能体针对模拟业务后端执行后，评估器将记录的工具参数和后端审计日志与该策略知识库进行比较。评估覆盖了九个广泛使用的智能体，以刻画基于目的的隐私过度披露。结果显示，成功的工具执行并不保证合理的隐私披露：智能体可能在完成任务的同时，通过中间工具调用传输不必要的私有信息。因此，ToolPrivacyBench形式化了按需知情的披露边界，据此每个工具应仅接收为其既定目的所必需的信息，并通过轨迹级审计识别多工具工作流中的隐私过度披露。

## Abstract
Large language models (LLMs) have increasingly moved from standalone text generation systems to agents that invoke external tools, access environments, and execute multi-step tasks. However, conventional function-calling benchmarks mainly evaluate task completion and API correctness, while privacy evaluation benchmarks typically focus on final responses or privacy judgments. Neither perspective captures purpose-bound information flow across an executed multi-tool trajectory. Motivated by this limitation in current agent evaluation, ToolPrivacyBench audits whether task-private atoms are routed only to authorized tools and downstream sinks, thereby evaluating both task completion and privacy over-disclosure during tool use. The benchmark contains 2,150 cases, including 1,150 fully synthetic privacy-sensitive business workflows and 1,000 cases adapted from existing multi-tool and function-calling benchmarks. Each case is represented by a policy knowledge base. After an agent executes against mock business backends, the evaluator compares recorded tool arguments and backend audit logs with this policy knowledge base. The evaluation covers nine widely used agents to characterize purpose-bound privacy over-disclosure. The results show that successful tool execution does not imply appropriate privacy disclosure: an agent may complete a task while transmitting unnecessary private information through intermediate tool calls. ToolPrivacyBench therefore formalizes a need-to-know disclosure boundary, under which each tool should receive only the information necessary for its stated purpose, and uses trajectory-level auditing to identify privacy over-disclosure in multi-tool workflows.

### 2. 论文提出的方法论 (核心思想、技术细节、算法流程)

论文的核心思想是构建一个**基于知识审计的基准**，用于衡量工具型LLM智能体是否遵循了**按需知情 (Need-to-Know)** 的披露原则。

*   **核心思想**:
    *   **基于目的的隐私 (Purpose-Bound Privacy)**：同一私有信息字段 (如“症状”) 在一个工作流的不同阶段 (如“临床记录” vs. “账单支付”) 拥有不同的授权状态。隐私风险并非取决于字段本身是否敏感，而是取决于它是否被发送给了不必要的工具。
    *   **轨迹级审计 (Trajectory-Level Auditing)**：评估不只看最终输出，而是通过记录完整的工具调用轨迹和后端审计日志，来检测中间过程的信息泄露。

*   **关键技术细节与工作流程 (评价管线)**:
    1.  **策略知识库构建 (Policy Knowledge Base)**
        *   **案例表示**: 每个案例 `c = (x, T, P, A)` 包含用户任务 (`x`)、工具集 (`T`)、私有原子集 (`P`) 和字段-工具授权矩阵 (`A`)。
        *   **私有原子 (Private Atoms, `P`)**: 将用户任务中的私有信息（如姓名、银行卡号、家庭住址）细化为可追踪的信息单元。
        *   **工具、目的与信息接收方 (Tools, Purposes, Sinks)**: 每个工具 `ti` 都明确定义了其在工作流中的目的 (`purpose_i`)、输入模式 (`schema_i`) 以及信息接收方类型 (`sink_i`)，如后台记录、工单系统、通知、交接消息等。
        *   **授权矩阵 (`A`)**: 一个 `|P| × |T|` 的矩阵，`A_{j,i} = 1` 表示私有原子 `p_j` 对于工具 `ti` 在当前任务目的中是必要的（已授权），`A_{j,i} = 0` 则表示不应被披露（禁止）。
    2.  **智能体执行与轨迹捕获 (Agent Execution & Trajectory Capture)**
        *   智能体在模拟业务后端环境中执行任务，所有工具调用产生的参数和与后端的交互都被记录为审计日志。
        *   生成的轨迹 `τ` 包含每一步调用的工具和传入的完整参数，包括结构化的字段和自由文本字段。
    3.  **隐私审计引擎 (Privacy Audit Engine)**
        *   **披露检测器**: 在工具参数（包括消息、摘要、备注等自由文本字段）中检测是否存在私有原子 `p_j`。检测支持精确匹配、别名和语义变体。
        *   **授权推理器**: 将检测到的“私有原子-工具”对与策略知识库中的授权矩阵进行比对。
        *   **过度披露判定**: 对轨迹中的第 `k` 步调用，若私有原子 `p_j` 出现在参数 `a_k` 中，但该原子对调用的工具 `t_{i(k)}` 是未授权的 (`A_{j,i(k)} = 0`)，则判定为一次基于目的的过度披露 `O_{j,k} = 1`。

### 6. 论文的主要结论与发现

1.  **任务成功与隐私合规脱节**: 在合成数据集上，虽然大部分模型的 TaskSuccess 很高 (92% - 97%)，但 MT-POI 也高达 19.19 - 28.04。这表明模型能有效完成任务，但同时存在严重的隐私过度披露。

4.  **泄露具有传播性**: 工作流路径分析显示，一旦私有信息在一次调用中被泄露，它很可能会沿着后续的“记录→工单”、“文档→通知”等转换路径继续传播。
5.  **效用-隐私权衡**: 不同的模型在效用和隐私之间存在不同的权衡模式。一些模型（如Qwen3.6-plus, Kimi K2.5）任务完成度高但隐私保护弱，而另一些（如Gemini 3.5 Flash）则相反。

### 8. 不足与局限

*   **模拟环境偏差**: 评估是在模拟业务后端和合成的私有信息上进行的，结果可能无法完全代表在生产环境中，面对真实用户数据和复杂业务逻辑时的隐私表现。
*   **策略知识库的静态性与成本**: 基准依赖于预定义且标注完善的策略知识库，这在实际应用中构建和动态维护成本很高。
*   **对抗性风险的排除**: 基准明确排除了对抗性攻击（如提示注入、越狱），仅关注良性任务中的非恶意泄露，因此未覆盖攻击者操纵下的隐私风险。
*   **基准的限定性**: 策略知识库只用作审计，并未提供给智能体作为防御手段。因此，评估的是模型在没有明确的运行时隐私保护机制下的“裸”行为。
*   **信息泄露检测的局限**: 尽管检测涵盖了别名和语义变体，但面对更复杂的、隐含的或通过编码表达的隐私泄露，自动化检测仍可能存在疏漏。
